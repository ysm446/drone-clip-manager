# スクリーンショット機能（F9 / F12）

作成日時: 2026-07-13 04:05
更新日時: 2026-07-13 04:05

F9 / F12 の 2 系統のスクリーンショットの仕様・実装・設計判断のメモ。

## 2 系統の違い

| | F9（動画フレーム保存） | F12（アプリ画面保存） |
|---|---|---|
| 内容 | 再生中動画の実フレーム | アプリ画面全体（mpv の映像は縮小合成） |
| 解像度 | **元動画のフル解像度**（4K 素材なら 3840×2160） | ウィンドウ解像度 ×表示スケール（例 1600×900 の 150% → 2400×1350） |
| 取得方法 | mpv の `screenshot-to-file`（mpv 不使用 / 失敗時は ffmpeg でフレーム抽出にフォールバック） | Chromium のページキャプチャ + mpv 現フレームの合成 |
| 保存先 | ライブラリ直下 `screenshots/` に `<動画名>_<位置>.png` | 同 `screenshots/` に `app_<日時>.png` |
| 用途 | 素材の静止画切り出し（品質優先） | 画面の記録・共有（UI ドキュメント用） |

## ファイル容量について（F9 の方が大きい理由）

1. **画素数**: F9 は元動画のフル解像度（4K = 約 830 万画素）。F12 はウィンドウ解像度
   （約 320 万画素程度）で、映像部分も縮小されて写る。
2. **絵柄と PNG の相性**: どちらも PNG（可逆）だが、ドローン映像のような自然画は
   ディテール・ノイズが多く可逆圧縮が効きにくい（1 枚 10〜20MB 級もあり得る）。
   F12 はダークテーマの平坦な UI が大半で、PNG が非常によく効く。

→ 仕様として妥当（F9 は品質優先のフル解像度）。容量を抑えたい用途が出てきたら、
F9 を JPEG（品質 90 前後で 1/10 程度）にする、または保存形式を選べるようにする拡張が候補。

## 実装ポイント

- [src/main/services/screenshot.ts](../../src/main/services/screenshot.ts)
  - `captureScreenshot(videoRelPath, timeSec, useMpv)` — F9 本体。mpv 優先 + ffmpeg フォールバック。
    出力名は `<動画名>_<00m34s801>.png` 形式、既存があれば連番。
  - `saveAppScreenshot(buf)` — F12 の保存（レンダラで合成した PNG バイト列を受け取る）。
- F12 の合成はレンダラ側: `capturePageDataUrl()`（Chromium 層。mpv 映像は写らない）と
  `mpvFrameDataUrl()`（mpv の現フレーム）を取得し、canvas で動画矩形に合成してから
  `saveAppScreenshot` に渡す。mpv がネイティブ最前面ウィンドウでページキャプチャに
  写らないための構成（docs/plan/progress.md の mpv 埋め込みの経緯も参照）。
- キーリピート / 二重イベントによる多重発火は App 側でタイムスタンプでガード。
