# plan — drone-clip-manager 実装方針と予定

作成日時: 2026-07-08 12:29
更新日時: 2026-07-17 15:58

実装方針・優先順位・今後の予定をまとめる。詳細仕様は [../spec.md](../spec.md)、目的は [goals.md](goals.md) を参照。

## アーキテクチャ方針

```
Renderer (React)        Library / Editor / Export Queue、<video> プレビュー
      │ IPC (invoke/handle)
Main Process (Node)     MediaService / ThumbService / ExportService / DB(better-sqlite3)
      │ child_process
ffmpeg / ffprobe
```

- **バックエンドサイドカーは持たない**。Electron メインプロセスから直接 `ffmpeg` / `ffprobe` を叩く。
- 将来のカメラモーション解析（Phase 3）は、メインプロセス下に `AnalysisService`（Python サイドカー）を差し込む形にする。
  今回は接続点（DB の `motion_tags` 受け皿、サービス分離）だけ用意し、実装しない。

## ストレージ / メタデータ配置

- ルートフォルダは **ユーザーがダイアログで指定**（ハードコードしない）。設定に永続化し次回復元。複数ルート切替可。
- メタデータは **ルート直下の `.dcm/`** に作成する:
  - `library.db`（SQLite: `videos` / `keyframes` / `segments`）
  - `thumbnails/`（サムネイルキャッシュ）
  - `proxies/`（将来のプレビュープロキシ）
- `videos.path` / `thumb_path` は **ルート相対パス** で保持し、フォルダ移動に耐える。

## 技術的な肝: キーフレームとロスレス切り出し

- キーフレーム抽出は ffprobe のパケットフラグから高速取得し、取り込み時に `keyframes` へキャッシュ。
- スナップ: **in は直前キーフレーム / out は直後キーフレーム**（区間が縮まないよう外側へ丸める）。UI にマーカー表示。
- 書き出し: `ffmpeg -ss <in_snapped> -i input -t <duration> -c copy -map 0 -avoid_negative_ts make_zero out.mp4`
  - `-ss` は `-i` の前（input seeking）。区間長は `-to` ではなく `-t`。音声も copy。
- ExportService は `mode: "keyframe" | "smart"` を切替可能なIFにしておく。今回は `keyframe` のみ実装。

## 開発フェーズと優先順位

### Phase 1 — ライブラリ基盤
- ルートフォルダ指定・永続化、`.dcm/` 初期化、DB スキーマ作成
- ルート配下の再帰走査で動画検出、ffprobe メタ抽出、DB 登録
- サムネイル生成（生成タイミングは §オープン論点）
- Library View（グリッド/リスト、バッジ、フィルタ、検索）
- プレビュー再生（`<video>`）

### Phase 2 — 区間 & 書き出し（← MVP ゴール）
- キーフレーム抽出・タイムライン可視化
- 区間ブックマーク CRUD（複数・ラベル・メモ・色）
- in/out スナップ（前/後キーフレーム）
- ロスレス書き出し（stream copy、単発 / バッチ）
- Export Queue（出力先、命名テンプレート、進捗、成功/失敗ログ）

### Phase 2.5 — クリップ一覧ビュー（区間を独立したクリップとして扱う）✅ 完了（2026-07-08）

区間（segment）を「動画に紐づくブックマーク」から一歩進めて、**独立したクリップとして横断一覧**する
ビューを追加する。名前どおりの "clip manager" にする中核機能。

- **狙い**: 全動画をまたいで「使えるカット」を俯瞰・整理・書き出しできるようにする。Export の
  「複数動画横断キュー」（spec §7.3）もこのビューから素直に実現する。
- **表示切替**: ライブラリ（動画一覧）⟷ クリップ（区間一覧）をトグル/タブで切替。
- **クリップカード**（グリッド or リスト）: in 点サムネイル / 元動画名 / ラベル / `in–out`・長さ /
  コーデック・解像度バッジ。
- **アクション**: クリック → 元動画を開いて in へシーク（範囲ループ再生も検討）／ラベル編集 ／
  書き出し ／ 削除。
- **複数選択 → まとめて書き出し**（動画横断）。既存の Export モーダルをクリップ選択ベースに拡張。
- **絞り込み/ソート**: 元動画・日付・ラベル（将来: モーションタグ）。
- **実装メモ**:
  - データは `segments` にあるので、全区間を集める IPC（例 `segments:listAll` 結合クエリ）を追加。
  - クリップサムネイル: in 点のフレームを ffmpeg で 1 枚生成して `.dcm/thumbnails/` にキャッシュ。
  - 既存の区間リスト（Editor 内）とロジックを共有できる形にする。

### Phase 2.6 — クリップのシーケンス（つないで連続再生 → 連結書き出し）★次の主要機能
クリップを並べて/つないで、**1 本のシーケンス（ラフカット）として連続再生**し、そのまま
**再エンコードなしで連結書き出し**する。

- **段階**:
  1. **線形シーケンス（まず本命）**: クリップを順に並べる（ドラッグ並べ替え）→ 連続再生 → concat 書き出し。
  2. **ノードグラフ**: クリップをノードとして線でつなぎ、順路を組んで再生。分岐は発展として後回し。
- **進捗（2026-07-09）**: ユーザー要望により **2. ノードグラフから着手し、一本道・再生までを実装済み**
  （「シーケンス」タブ）。次は順路の **無劣化連結書き出し（concat）**。分岐と線形専用 UI は後続。
- **再生**: mpv のプレイリスト（`loadfile ... append` + 各クリップの in/out）で順再生。クリップ境界の
  ギャップレス度は要検証。
- **連結書き出し**: 同一コーデック/パラメータ（DJI 素材）なら **ffmpeg concat demuxer で stream copy 連結**
  （無劣化）。パラメータ差があるものは対象外 or 将来 smart 連結。spec §9「跨ぎ書き出し」がここに繋がる。
  → **実装済み（2026-07-12）**。2 段階 stream copy（切り出し → concat）+ 進捗モーダル。
- **論点**: 境界の滑らかさ、分岐ノードの再生ルート選択 UX（線形なら不要）。

#### Phase 2.6b — 連結書き出しに BGM を載せる ★予定（仕様確定・実装待ち / 2026-07-12）

連結書き出しのオプションとして BGM を 1 曲指定し、フェード付きで音声に載せる。
「映像はロスレスのまま」という設計思想は崩さない（copy 維持、音声のみエンコード）。

- **仕様**:
  - BGM フォルダ（既存の BGM プレイヤーと同じ `getBgm()` の曲一覧）から 1 曲選択。「なし」も可（従来どおり）。
  - 音声は **BGM で上書き**（元音声は出力に含めない。ドローン素材は元音声が無い/不要なため）。
  - **フェードイン / フェードアウト秒数**を設定可能（0 で無効）。
  - 映像は stream copy（無劣化）のまま。音声のみ AAC（256k 目安）にエンコード。
  - 曲がシーケンスより**長い場合は末尾で切る**。**短い場合は無音のまま**終わる（ループはしない）。
- **実装方針**:
  - `exportConcat` に `bgm?: { relPath, fadeInSec, fadeOutSec }` を追加。concat 結果を一時ファイルに出し、
    3 段目で BGM を合成して最終出力（進捗は cut / concat / bgm の 3 段階、総秒数×3 が分母）。
  - ffmpeg: `-i concat.mp4 -i music -map 0:v -map 1:a -c:v copy -c:a aac -b:a 256k
    -af "apad=whole_dur=T[,afade=t=in:st=0:d=fi][,afade=t=out:st=T-fo:d=fo]" -t T`
    （`apad` で曲が短い場合に無音を延長し、`-t` で総尺 T に切る。BGM パスは `resolveInBgm()` で解決）。
  - UI: 「連結書き出し…」押下 → 設定モーダル（BGM 選択 / フェードイン / フェードアウト）→
    出力先を選んで開始。既存の進捗 / 結果モーダルに続く。
  - 型: `ConcatBgm` を追加、`exportSequenceConcat(items, outDir, name, bgm?)`、
    `ConcatProgress.phase` に `'bgm'` を追加。
- **論点**: フェードの既定値（案: イン 1s / アウト 2s）、曲の開始オフセット指定（サビ合わせ）は将来。

### Phase 2.7 — キーフレームフラグと区間スナップ（マーキング補助）★予定
動画を見ながら**ショートカットでキーフレームにフラグ（マーカー）を立て**、そのフラグを基準に区間を張る
ワークフロー。区間の端（タイムラインの区間バー＝「テープ」）はフラグへ磁石スナップする。

- **フラグ設置**: 再生位置でショートカット（例 `F` / `M`）を押すと、その位置を**最寄り（または直前）の
  キーフレームにスナップしてフラグを立てる**。もう一度押す/別キーで解除。フラグはキーフレーム上にあるため、
  区間がフラグに揃えば**ロスレス書き出しの整合が保たれる**（フラグ = ユーザーが選んだキーフレームの部分集合）。
- **表示**: タイムラインにフラグ専用マーカー（キーフレーム tick とは別色/形）で表示。ホバーで時刻。
- **区間を張る**:
  - フラグを基準に区間を作る（例: 直近 2 フラグ、または「前フラグ → 次フラグ」）。
  - 区間バーの端ドラッグ / 移動時に、近傍フラグへ**磁石スナップ**（フラグが無い箇所は従来のキーフレーム
    スナップ in=前 / out=後）。
- **永続化**: フラグを動画単位で保存（新テーブル `flags(video_rel_path, pts_time)` など）。
- **論点**:
  - フラグのスナップは「最寄りキーフレーム」か「直前キーフレーム」か。
  - 区間生成のキー操作（フラグ 2 点指定 / in・out を個別にフラグへ）と既存 `I` / `O` の役割分担。
  - スナップの磁石半径（px / 秒）、スナップ一時無効の修飾キー（Alt でフリー移動など）。

### Phase 2.8 — ユーザーラベル / タグ（分類・絞り込み）★予定
動画・クリップに**ユーザー定義のラベル（タグ）**を付けて分類・検索・絞り込みできるようにする。
地名・時間帯（朝 / 昼 / 夕方 / 夜）・被写体・天候などを、後から自由に付与する。

- **対象**: 動画（フライト単位）と クリップ（区間）。両方に付けられるのが理想（要判断: まず区間から）。
- **タグの形**: 自由記述タグ + 任意で「カテゴリ（名前空間）」。例 `地名:ツェルマット` / `時間帯:夕方`。
  カテゴリ無しの素のタグも可。既存の `segment.label`（1 つの見出し）とは別に、**複数タグ**を持てる。
- **UI**: クリップカード / 動画メタバーにタグチップを表示・追加・削除。タグ入力は既存タグの補完付き。
  クリップビュー / ライブラリの絞り込みにタグ条件（AND / OR）を追加。
- **時間帯の補助**: `videos.recorded_at`（撮影時刻・取得済み）から時間帯を**自動サジェスト**できる（手動上書き可）。
- **永続化**: 新テーブル `tags(id, name, category)` + 中間表 `item_tags(tag_id, target_type, target_id)`
  （`target_type = 'video' | 'segment'`）。または簡易に `labels(target_type, target_id, key, value)`。
- **論点**:
  - まず区間（クリップ）だけにするか、動画にも付けるか。
  - カテゴリ（名前空間）を第一級にするか、`key:value` 記法の運用に留めるか。
  - 将来のモーションタグ（Phase 3 `motion_tags`）との統合（自動タグ + 手動タグを同じ土俵に）。

### Phase 2.9 — クリップ並び順の自動提案（構成アルゴリズム）★構想（2026-07-17）
[../reference/drone-editing-tips.md](../reference/drone-editing-tips.md) の「並べ方の 7 原則」を
スコア関数に落とし、**LLM ではなく決定的なアルゴリズム**でシーケンスの並び順を提案する。
設計の詳細は [../reference/clip-ordering-algorithm.md](../reference/clip-ordering-algorithm.md) を参照。

- **方式**: 遷移コスト付きの並べ替え最適化（章分け → 貪欲 + 2-opt → 冒頭/締めの特別処理）。
  決定的・説明可能・オフラインで、サイドカーなしの最小構成を崩さない。
- **提案は下書き**: 新しいシーケンスとして生成し、ノードグラフ上でユーザーが微調整。
  各接続に理由チップ（「同方向」「高度が下がる」等）を表示。
- **構成リントと表裏一体**: スコア関数を共通実装にし、「既存の並びの違反検出（リント）」と
  「違反が少ない並びの生成（提案）」の両方に使う。
- **依存**: Phase 2.8 の名前空間付きタグ（`motion:` / `dir:` / `scale:` / `speed:` / `alt:`）が
  実質の前提。クリップの評価（星 or `見せ場` タグ）は ⑦（冒頭に最高のカット）に必要で要検討。
- **段階**: v1 スコア + 2-opt（現状タグの `key:value` 運用で開始可）→ v2 `recorded_at` 章分け +
  評価 → v3 GPS 章分け（Phase 3 解析枠）。詳細は設計メモの「段階的実装案」。

### Phase 3 — ドローン強化 & 拡張
- `flight_group` 推定・グルーピング表示
- （任意）smart-cut（先頭/末尾 GOP のみ再エンコード）
- （任意）カメラモーション解析サイドカー（Python / OpenCV / RAFT / VLM）
- （任意）**シーン検出でフィルムストリップの抽出点を差し替え**（2026-07-10 メモ）:
  現状のフィルムストリップ（`Filmstrip.tsx`）は動画全体の 16 枚等間隔サンプリング。これを
  「変化のあるところ」の代表フレームに置き換える。`Filmstrip` は抽出時刻リスト（`times`）の
  算出を差し替えるだけで対応できる構造にしてある。
  - 実装候補: ffmpeg `select='gt(scene,閾値)'`。ただしドローン映像はカット切替のない連続ショットで
    シーン検出が反応しにくく、全編デコードが必要（4K HEVC で数分）。
  - 方針: 前景でやらず**バックグラウンド解析 + DB キャッシュ**（新テーブル例: `scene_marks(video_rel_path, pts_time, score)`）。
    カメラモーション解析サイドカーと同じ `AnalysisService` の枠に載せる。閾値ではなく
    「スコア上位 N 点 + 最低間隔」で選ぶ方がドローン映像には合いそう。
- （任意）分割ファイルを跨いだ連結切り出し（Phase 2.6 のシーケンス連結書き出しと統合）

## オープンな設計判断（着手前に詰めたい）

1. **HEVC 10bit / D-Log のプレビュー再生**: OS の HEVC 対応に依存するか、取り込み時に低解像度プロキシを生成するか。
   → プロキシ生成をデフォルト ON にするか要判断（実務上の最大の落とし穴）。
2. **キーフレームスナップの丸め**: in=前 / out=後 で確定するか、最寄りキーフレームにするか。
3. **flight_group 推定ロジック**: 連番 + 撮影時刻連続 + 同一解像度/fps のどれを主キーにするか。
4. **サムネイル生成タイミング**: 取り込み時バッチか、表示時遅延生成 + キャッシュか。
5. **ffmpeg バイナリ**: システム依存か同梱か（配布と HEVC 対応の安定性）。
6. **クリップモード（クリップ一覧タブ）の存廃**（2026-07-12 起票）:
   シーケンス画面のクリップエリアが育った結果（クリック再生 = クリップ範囲バー + ループ、
   in/out ナッジ、タグ絞り込み・検索、選択強調）、クリップモードと役割がかなり重複してきた。
   - **現時点でクリップモードにしかないもの**: ①複数選択 → 動画横断のまとめて書き出し、
     ②カード上のラベル編集・タグ付与/削除、③ソート（動画順/新しい順/長い順/ラベル順）と
     元動画絞り込み、④カード単位の書き出し/削除。
   - **統合案**: ①〜③をシーケンス画面のクリップエリアへ吸収し、タブを「ライブラリ /
     シーケンス」の 2 つにする。画面を跨ぐ「編集ジャンプ」系の機能も不要になる。
   - **保留の理由**: シーケンスを介さない素材整理（区間マーク → 個別書き出し = 当初 MVP）の
     使い方ではクリップモードが横断管理のハブ。パレットに管理機能を全部載せると重くなる
     トレードオフもある。
   - **判断基準**: 「クリップモードを開くのは編集ジャンプで飛ばされたときだけ」という使い方が
     続くようなら統合に踏み切る。着手時は移行設計を本ファイルに書いてから。
