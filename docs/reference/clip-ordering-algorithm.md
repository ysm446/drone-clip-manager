# クリップ並び順の自動提案 — アルゴリズム設計

作成日時: 2026-07-17 15:58
更新日時: 2026-07-17 15:58

[drone-editing-tips.md](drone-editing-tips.md) の「並べ方の 7 原則」を、
LLM ではなく **決定的なアルゴリズム** でシーケンスの並び順提案に落とすための設計メモ。

## なぜ LLM ではなくアルゴリズムか

検討の経緯（2026-07-17）:

- 7 原則の大半は「タグ + 秒数 + 撮影日時」に対する機械的なルールとして表現できる。
- アルゴリズムは **決定的・説明可能・オフライン・依存ゼロ** で、
  「バックエンドサイドカーを持たない最小構成」という本アプリの方針に合う。
- LLM（ローカル含む）の固有価値は「ラベルの意味理解」（地名から物語的つながりを読む）
  だけだが、撮影日時や GPS による章分けがあればほぼ代替できる。
- LLM は存在しないクリップ ID の混入・取りこぼしの検証が必要になり、
  本数が増える（50 本超）と順序の一貫性も崩れやすい。

結論: **まずアルゴリズム実装**。LLM は将来の「あれば使う」オプションに留める。

## 問題の定式化

クリップ列 `c1, c2, …, cn` の並びに対するスコアを定義し、スコア最大の並びを探す
**遷移コスト付きの並べ替え最適化**。

```
スコア(並び) = Σ 隣接ペアの遷移スコア(ci, ci+1)
             + 全体構造スコア(並び全体)
```

### 原則とスコア項の対応

| 原則 | スコア項 | 種別 | 必要データ |
|---|---|---|---|
| ② 方向を揃える | `dir:` が反転したらペナルティ（章の切れ目は免除） | 隣接ペア | `dir:` タグ |
| ④ 動きを続けない | 同じ `motion:` が 3 連続でペナルティ | 窓（3 個） | `motion:` タグ |
| ⑤ 速度のリズム | `speed:` の波（遅→速→遅…）に沿えばボーナス | 系列 | `speed:` タグ |
| ③ 高度を下げていく | `alt:` がおおむね単調減少ならボーナス + 最後が上昇なら締めボーナス | 全体 | `alt:` タグ |
| ① 大→小→大 | `scale:` が 遠景→寄り→遠景 の折り返しに沿えばボーナス | 全体 | `scale:` タグ |
| ⑦ 冒頭に最高のカット | 評価最高のクリップが先頭にあればボーナス | 全体 | 星評価 or `見せ場` タグ |
| ⑥ 旅を作る | 章分け（後述）で骨格として扱う。スコアではなく制約 | 前処理 | `recorded_at` / GPS |
| 秒数の目安（§4） | 4 秒未満 / 15 秒超のクリップにペナルティ（並びではなく素材への警告） | 単項 | 秒数（取得済み） |

タグは Phase 2.8 の名前空間付きタグ（`motion:orbit` 等）を前提とする
（[drone-editing-tips.md](drone-editing-tips.md) §6.1 の語彙）。

## 解法（3 ステップ）

対象は数十〜100 本程度を想定。厳密解（TSP 相当で NP 困難）は不要で、
ヒューリスティクスで十分良い並びが得られる。

### Step 1 — 章分け（旅の骨格）

- `videos.recorded_at`（取得済み）で時系列に並べ、時間差が大きいところで章に分割
  （単純なギャップ閾値クラスタリング）。
- 将来: DJI 素材の GPS メタデータ（SRT / メタデータ内）が使えれば
  「撮影位置でクラスタリング → 移動順に章を並べる」に置き換える。
  地名ラベルから LLM に推測させるより正確。
- 章の切れ目では ② 方向ペナルティを免除する（場面転換の逆方向は効果的、の原則どおり）。

### Step 2 — 章内の並べ替え

1. **貪欲法で初期解**: 先頭から「次に置くと遷移スコアが最大のクリップ」を順に選ぶ。
2. **2-opt（山登り）で改善**: 2 点スワップ / 区間反転を試し、スコアが上がれば採用。
   改善がなくなるまで繰り返す。ビームサーチ（幅 5〜10）でも可。
3. 決定的にするため、同点時のタイブレークはクリップ ID 順など固定順で行う。

計算量の目安: n ≤ 100 なら 2-opt 数千イテレーションでもミリ秒〜数秒。

### Step 3 — 冒頭と締めの処理

- 評価最高（星評価 or `見せ場` タグ）のクリップを先頭へ（⑦）。
- 冒頭の合計が約 15 秒に収まるよう先頭ブロックを構成。
- 末尾は `alt:` 上昇 or `scale:遠景` のクリップを優先（締めの上昇 / 引き）。

## 出力と UX

- **提案はあくまで下書き**。新しいシーケンスとして生成（既存を上書きしない）し、
  ユーザーがノードグラフ上で微調整する。
- 各接続に **理由チップ** を表示できる（「同方向」「高度が下がる」「動きが変わる」）。
  スコア項がそのまま説明になるのがアルゴリズムの強み。
- 構成リント（[drone-editing-tips.md](drone-editing-tips.md) §6.2）と表裏一体:
  リントは「既存の並びの違反検出」、本機能は「違反が少ない並びの生成」。
  **スコア関数を共通実装** にすれば両方に使える。

## 欠損データの扱い

- タグが無いクリップは該当スコア項を **中立（0）** として扱い、提案結果に
  「タグ不足で精度が下がっている」旨を表示する。
- 全クリップにタグが無い場合でも、秒数リズム + 撮影時刻順（章分け）だけで
  最低限の提案は成立する。

## 前提条件と依存

1. **Phase 2.8 のカテゴリ（名前空間）付きタグ** — `motion:` / `dir:` / `scale:` /
   `speed:` / `alt:` の語彙。現状の自由タグでも `key:value` 記法の運用で開始可能。
2. **クリップの評価（⑦ 用）** — 星評価 or `見せ場` タグ。未実装（要検討）。
3. `videos.recorded_at` — 取得済み。
4. GPS メタデータ抽出 — 将来（Phase 3 の解析枠）。

## Phase 3 との関係

カメラモーション自動解析が `motion:` / `dir:` / `speed:` タグを自動付与できるように
なれば、本アルゴリズムは手作業のタグ付けなしで機能する。
